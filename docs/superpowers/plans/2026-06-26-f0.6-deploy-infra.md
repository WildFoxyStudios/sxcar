# F0.6 — Deploy & Infra · Plan de ejecución

Spec: `docs/superpowers/specs/2026-06-26-f0.6-deploy-infra-design.md`
Branch: `f0.6-deploy-infra` (desde `main` @ 02f1990)
Modo: mezcla — IaC/MCP los ejecuto yo con verificación; el `apply` de AWS y la creación de Neon esperan a que el usuario rellene `.env`/`tfvars`.

## Principios
- 0 secretos en git. Plantillas `*.example` commiteadas; reales gitignored.
- Nada destructivo sobre recursos existentes del usuario. Solo crear `proyectox-*`.
- Cada recurso vivo creado se verifica (list/get) y se documenta cómo revertir.

## Tareas

### T1 — Scaffolding `infra/` + gitignore + plantillas + runbook
- `infra/README.md` (runbook: orden de provisión, qué rellenar, cómo revertir).
- `infra/.env.example` (DATABASE_URL, DATABASE_URL_DIRECT, JWT_SECRET, REDIS_URL, R2_*). 
- `infra/terraform/` con `versions.tf`, `variables.tf`, `main.tf`, `outputs.tf`, `terraform.tfvars.example`.
- Raíz `.gitignore`: `**/.env`, `!**/.env.example`, `*.auto.tfvars`, `infra/terraform/*.tfvars` (no example), `*.tfstate*`, `.terraform/`, `.vercel/`.
- **Acept.:** `git status` no muestra secretos; `terraform fmt -check` ok; example files completos.

### T2 — Cloudflare R2 (vía MCP, lo ejecuto yo)
- Crear `proyectox-media`, `proyectox-private`, `proyectox-verification`.
- Documentar en runbook: crear R2 API Token (Object R/W, scoped a esos 3 buckets) → rellenar `R2_*` en `.env`. CORS para subida directa (JSON de CORS en `infra/r2-cors.json`).
- **Acept.:** `r2_buckets_list` muestra los 3 nuevos; los 4 viejos intactos.

### T3 — Terraform AWS (autor + plan; apply diferido)
- VPC por defecto o mínima; `aws_security_group` (22 desde `admin_cidr`, 80/443 público); `aws_instance` t4g.small, AMI Ubuntu 24.04 arm64 (data source SSM), key pair, EBS gp3.
- `user_data`: instalar Docker + compose, crear `/opt/proyectox`, escribir `docker-compose.prod.yml`, plantilla `.env` (vacía; se rellena por SSH/SSM), `caddy` + `redis` + `api`. NO hornear secretos.
- `outputs`: IP/DNS público, comando SSH.
- **Acept.:** `terraform init` + `terraform validate` + `fmt` ok; `terraform plan` corre (con creds del usuario) sin error. `apply` cuando el usuario confirme.

### T4 — `docker-compose.prod.yml` + build ARM
- Servicios: `caddy` (reverse proxy, :80/:443), `api` (imagen arm64), `redis:7-alpine` (solo interno).
- Caddyfile: dominio→auto-HTTPS; staging temporal→`:80` reverse_proxy a api:8080 (HTTP).
- Script `infra/build-and-push-arm.sh` (buildx arm64 → registry; o build en la instancia). Decidir registry: GHCR (gratis) o build local en EC2.
- **Acept.:** compose válido (`docker compose config`); imagen arm64 construible.

### T5 — Vercel marketing (vía MCP, lo ejecuto yo)
- `apps/marketing/vercel.json` (o `vercel.ts`) con `buildCommand: "npm run typecheck && npm run build"`, framework nextjs, root.
- Crear proyecto `proyectox-marketing` (team del usuario) + deploy de producción. Capturar URL `.vercel.app`.
- **Acept.:** deploy READY; la URL sirve la landing; tags SEO presentes (canonical/OG/JSON-LD); typecheck corrió en build.

### T6 — Neon (runbook + migraciones; el usuario crea el proyecto)
- Runbook: crear proyecto Neon `proyectox` (región aws-us-east), copiar string pooled + directo al `.env`, habilitar PostGIS.
- Una vez relleno: `DATABASE_URL=<directo> api migrate` (o `sqlx migrate run`) → 10 migraciones; smoke `api` `/health` apuntando a Neon (pooled) localmente.
- **Acept.:** migraciones aplicadas en Neon; `/health` ok contra Neon.

### T7 — CI (GitHub Actions)
- `.github/workflows/marketing.yml`: `npm ci && npm run typecheck && npm run lint && npm test && npm run build` (paths: apps/marketing).
- (Scaffold) `.github/workflows/backend-deploy.yml`: build arm64 → push GHCR → (gated) ssh/SSM deploy. Sin secretos hardcoded; usa GitHub Secrets.
- **Acept.:** workflow marketing válido (yaml lint) y verde en CI si hay remoto; backend-deploy escrito y documentado (deshabilitado hasta tener secrets).

### T8 — Smoke E2E + cierre
- Marketing live (curl tags SEO). API `/health` live (cuando EC2+Neon estén). Actualizar `site.url`/envs si procede.
- Commit, revisión final, integrar a `main`, actualizar ledger + memoria.
- **Acept.:** lo que esté live, verificado; lo diferido (apply AWS/Neon), claramente marcado como pendiente de credenciales.

## Orden de ejecución
T1 → T2 (vivo) → T5 (vivo) → T3/T4 (autor) → T6/T7 (runbook+CI) → esperar `.env` del usuario → apply AWS + Neon migrate + smoke → T8.

## Reversión
- R2: `r2_bucket_delete` de los `proyectox-*`. Vercel: borrar proyecto. AWS: `terraform destroy`. Neon: borrar proyecto en consola. Nada de esto toca recursos preexistentes.
