# Spec F0.2 — Esquema completo de Neon + sistema de migraciones

- **Proyecto:** proyecto-X
- **Fecha:** 2026-06-26
- **Fase:** F0.2 (deriva del §6 del spec maestro `2026-06-25-grindr-clone-design.md`)
- **Alcance (elegido por el usuario):** esquema **completo** (todas las tablas del §6) en una sola fase, vía migraciones agrupadas por dominio.

> Objetivo: dejar asentado el **modelo de datos entero** + el **sistema de migraciones** + el flujo de **queries verificadas en compilación** (`sqlx::query!` con cache offline), de modo que cada fase posterior solo añada lógica (repos/handlers) sobre tablas ya existentes.

---

## 1. Tooling y convenciones

- **Migraciones:** `sqlx-cli` con archivos SQL versionados en `backend/migrations/NNNN_nombre.sql` (solo "up"; al ser greenfield no necesitamos "down" por ahora — se documenta).
- **Ejecución:** un **subcomando `migrate`** del binario (`api migrate` o un binario `migrate` dedicado) que llama a `sqlx::migrate!()`. **No** se migra automáticamente en cada arranque del servidor (evita carreras entre instancias en producción). En tests, cada test aplica las migraciones a una DB limpia.
- **Compile-time checking:** `cargo sqlx prepare` genera el cache `.sqlx/` (commiteado). En CI/build, `SQLX_OFFLINE=true` permite compilar `query!` sin DB. El step de tests sí usa una DB (service Postgres) y aplica migraciones.
- **Convenciones de modelado:**
  - **PK:** `uuid` con `gen_random_uuid()` (pgcrypto) en todas las tablas con entidad propia (no-enumerables → mejor privacidad). `audit_log` usa `bigserial` (alto volumen, orden).
  - **Tiempos:** `timestamptz`; `created_at default now()`; `updated_at` mantenido por trigger `set_updated_at()`.
  - **Conjuntos cerrados** (status, role, provider, platform, kinds, position, tier…): **`text` + `CHECK`** (fáciles de evolucionar; la taxonomía "fuente de verdad" del producto vive en `packages/shared`).
  - **Geo:** `geography(Point,4326)` (PostGIS) + índices **GiST**.
  - **Borrado:** `users.deleted_at` (soft-delete) + FKs `ON DELETE CASCADE` en datos poseídos por el usuario, para que una purga dura (derecho al olvido GDPR) elimine en cascada.
  - **Datos de salud** (`profiles.hiv_status`, `last_tested_on`, `prep`): categoría especial GDPR; el **cifrado a nivel de campo se aplica en la capa de aplicación** (F1) — el esquema almacena el valor (texto/ciphertext). Se documenta.
  - **Bloqueos/visibilidad** se aplican en queries de la app (no en el esquema), pero `blocks` existe aquí.

## 2. Migraciones (8, agrupadas por dominio)

### `0001_extensions.sql`
- `CREATE EXTENSION IF NOT EXISTS postgis;`
- `CREATE EXTENSION IF NOT EXISTS pgcrypto;`  (uuid)
- `CREATE EXTENSION IF NOT EXISTS citext;`  (emails/tags case-insensitive)
- Función `set_updated_at()` (trigger genérico que pone `NEW.updated_at = now()`).

### `0002_identity.sql`

Trigger updated_at.
- **auth_identities**(`id` uuid PK, `user_id` uuid NN → users CASCADE, `provider` text CHECK in (apple,google,password), `provider_uid` text NN, `created_at`, UNIQUE(`provider`,`provider_uid`)). Index(`user_id`).

## 3. Capa `db` (Rust)

- `db::migrate(pool) -> Result<()>` envolviendo `sqlx::migrate!("./migrations")`.
- Subcomando `migrate` en el binario `api` (`api migrate`) que conecta y migra.
- **Repos núcleo** (con `query!` chequeado en compilación), suficientes para validar el flujo `sqlx prepare` y servir a F0.3:
  - `users`: `create_user`, `find_user_by_email`, `find_user_by_id`, `mark_email_verified`.
  - `auth_identities`: `add_auth_identity`, `find_by_provider`.
  - `consent_records`: `record_consent`.
  - (El resto de repos se implementa en su fase; F0.2 solo asienta el esquema + un repo núcleo representativo.)
- `.sqlx/` cache generado con `cargo sqlx prepare` y commiteado.

## 4. Testing

- **Migraciones:** test que crea una DB/esquema limpio, aplica `migrate()`, y asierta existencia de tablas/columnas/índices clave (consultando `information_schema` / `pg_indexes`). Re-`migrate()` es idempotente (no falla).
- **Repos:** tests de integración contra Postgres en vivo — `create_user` + `find_user_by_email` (round-trip), unicidad de email (violación esperada), `add_auth_identity`, `record_consent`.
- **PostGIS:** un test mínimo que inserta una `location` y hace un `GEOSEARCH`/`ST_DWithin` para confirmar que el tipo geography + índice GiST funcionan.
- Aislamiento: cada test usa un esquema temporal o una DB de test dedicada (estrategia a fijar en el plan; por defecto, esquema `test_<rand>` o `TRUNCATE` entre tests sobre la DB local 5433).

---

## 5. CI

- Añadir `SQLX_OFFLINE: "true"` al job para que `cargo build`/`clippy` compilen `query!` desde `.sqlx/`.
- El step de tests mantiene el service Postgres + `TEST_DATABASE_URL`; los tests aplican migraciones.
- (Opcional) un step que verifica que `.sqlx/` está actualizado (`cargo sqlx prepare --check`) para que nadie olvide regenerar el cache.

---

## 6. Riesgos / notas

- **Esquema completo de golpe** = más superficie ahora; mitigación: migraciones agrupadas y revisadas por dominio, y solo se implementan repos del núcleo (no de todo) en F0.2.
- Tablas de fases futuras (chat, álbumes, etc.) quedan creadas pero sin lógica — está bien (no rompen nada; se rellenan en su fase).
- `CHECK` en vez de enums nativos: se eligió por evolución; si una lista crece mucho, se considerará tabla de catálogo.
- Cifrado de campos de salud: **pendiente para F1** (capa app); el esquema no lo fuerza.
