# Spec F0.4 — React Native app shell (Expo)

- **Proyecto:** proyecto-X
- **Fecha:** 2026-06-26
- **Fase:** F0.4 (deriva del §3/§4 del spec maestro)
- **Alcance:** esqueleto de la app **Expo / React Native** (iOS + Android + Web vía react-native-web) con navegación, gestión de sesión y pantallas de **auth conectadas al backend** F0.3.

> Objetivo: una app que arranca, navega entre un grupo "auth" (login/register/verify) y un grupo "app" (placeholder tras login), con almacenamiento seguro de tokens, refresh automático y un cliente de API tipado contra `/auth/*`. Verificable por typecheck + lint + tests de componente + export web. El pulido visual y las features (grid, chat…) van en sus fases.

---

## 1. Decisiones de stack

- **Expo** (managed workflow) + **TypeScript** + **expo-router** (routing por archivos; unifica móvil y web).
- **TanStack Query** (estado de servidor) + **Zustand** (estado local / sesión).
- **expo-secure-store** para tokens en móvil; fallback a `localStorage` en web (helper de almacenamiento seguro multiplataforma).
- **Estilo:** sistema mínimo basado en **StyleSheet** + tokens de tema (colores, espaciado, tipografía). *Decisión de riesgo:* NativeWind/Tailwind se adopta en las fases de UI (menos superficie de tooling ahora; el shell prioriza arrancar y verificarse).
- **Código autocontenido en `apps/mobile/`** (sin imports desde `packages/*` todavía) para evitar fricción de Metro con el monorepo. La extracción a `packages/api-client`/`packages/ui` se hará en F0.5 cuando se comparta con la web.
- Config de API por **`EXPO_PUBLIC_API_URL`** (default dev: el backend local en `http://localhost:8081`).

## 2. Estructura (apps/mobile)

```
apps/mobile/
  app/                         # expo-router (rutas por archivo)
    _layout.tsx                # providers (QueryClient) + gating de auth (redirect)
    (auth)/_layout.tsx         # stack del grupo auth
    (auth)/login.tsx
    (auth)/register.tsx        # email, password, DOB (age-gate), consentimientos
    (auth)/verify-email.tsx
    (app)/_layout.tsx          # stack del grupo app (protegido)
    (app)/index.tsx            # placeholder "home" (tras login) + logout
  src/
    api/client.ts              # fetch wrapper: baseURL, auth header, 401→refresh→retry
    api/auth.ts                # register/login/refresh/verifyEmail/logout tipados
    api/types.ts               # tipos de request/response de auth
    auth/store.ts              # Zustand: tokens, status, signIn/signOut/hydrate
    auth/storage.ts            # secure-store (móvil) / localStorage (web)
    theme/tokens.ts            # colores, espaciado, tipografía
    ui/{Button,TextField,Screen,Text}.tsx  # componentes base (StyleSheet)
    config.ts                  # API base URL desde EXPO_PUBLIC_API_URL
  app.json / app.config.ts     # config Expo (name, scheme, web bundler)
  package.json, tsconfig.json, babel.config.js, metro.config.js
  jest setup + __tests__/      # tests de componente/lógica
```

## 3. Sesión & API

- **`auth/store.ts` (Zustand):** `{ status: 'loading'|'signedOut'|'signedIn', accessToken, refreshToken, signIn(pair), signOut(), hydrate() }`. `hydrate()` carga tokens del storage al iniciar. `signOut()` limpia storage y llama (best-effort) a `/auth/logout`.
- **`auth/storage.ts`:** `getTokens()/setTokens()/clear()` usando `expo-secure-store` en nativo y `localStorage` en web (detección de plataforma).
- **`api/client.ts`:** `request()` que añade `Authorization: Bearer <access>`; si recibe `401`, intenta `/auth/refresh` con el refresh token (rotación → guarda el par nuevo) y reintenta una vez; si el refresh falla → `signOut()`.
- **`api/auth.ts`:** `register({email,password,dob,consents})`, `login({email,password})`, `verifyEmail(code)`, `resendEmail()`, `logout(refresh)` — tipados con `api/types.ts`.

## 4. Pantallas (shell)

- **login:** email + password → `login()` → `signIn(pair)` → redirect a `(app)`. Errores genéricos.
- **register:** email, password (min 8), **fecha de nacimiento** (con validación client-side de **18+** que refleja el age-gate del servidor), checkboxes de consentimiento (ToS/privacidad/edad) → `register()` → `signIn` → redirect a verify-email.
- **verify-email:** input de código → `verifyEmail()`; botón "reenviar".
- **(app)/index:** placeholder ("Bienvenido / aquí irá el grid") + botón **logout**.
- **Gating:** `app/_layout.tsx` observa el store; `status==='signedIn'` → permite `(app)`, si no → redirige a `(auth)/login`. `status==='loading'` → splash mientras hidrata.

## 5. Verificación (qué significa "verde" en F0.4)

- **`tsc --noEmit`** sin errores (typecheck estricto).
- **ESLint** limpio (config Expo/TS).
- **Tests de componente/lógica (jest-expo + @testing-library/react-native):**
  - el store transiciona signedOut→signedIn→signedOut;
  - `api/client` añade el header y, ante 401, refresca y reintenta (con fetch mockeado);
  - login screen renderiza y dispara `login` al enviar;
  - register valida age-gate <18 (no envía) y ≥18 (envía);
  - el gating redirige según status.
- **`expo export -p web`** completa (bundle web OK).
- (E2E con Detox/Playwright y prueba contra el backend real: fases posteriores.)
