# Spec F0.5 — Sitio de marketing Next.js (SEO)

- **Proyecto:** proyecto-X
- **Fecha:** 2026-06-26
- **Fase:** F0.5 (deriva del §15 del spec maestro — "SEO superior")
- **Alcance:** el **sitio de marketing público** en Next.js (App Router) en `apps/marketing/`, optimizado para SEO (SSR/SSG/ISR), separado de la app de producto (que va detrás de login y es `noindex`).

> Objetivo: un sitio que renderiza HTML completo para crawlers, con el **núcleo SEO** (metadata, datos estructurados JSON-LD, sitemap, robots, canónicas, OG), una landing pulida + páginas legales/soporte, y un hub de contenido (blog) placeholder. Verificable por `next build` + lint + tests. La internacionalización (i18n/hreflang) queda **lista para añadir** y es un fast-follow.

---

## 1. Stack y decisiones

- **Next.js (App Router) + TypeScript + Tailwind CSS** (scaffold con `create-next-app`). Tailwind alinea con la futura adopción de NativeWind en la app.
- **Sin librería de UI** (shadcn/etc.): landing hecha a medida con Tailwind para un diseño distintivo (no plantilla genérica).
- **Idioma:** **inglés** primario (default global). Estructura lista para i18n (es y otros) como fast-follow — NO se monta `next-intl` con routing por locale en F0.5 para de-riesgar; se documenta el plan.
- **Despliegue:** el sitio se **construye** localmente (`next build`) en F0.5; el `vercel deploy` real se hace cuando conectes Vercel (F0.6 o cuando quieras) — necesita tu cuenta.
- **App autocontenida** en `apps/marketing/` (sin imports de `packages/*` aún; la extracción del design system compartido es posterior). Usa npm (como `apps/mobile`).

## 3. SEO (el núcleo de la fase)

- **Render:** páginas estáticas (SSG) por defecto → HTML completo para crawlers. Sin estado de cliente innecesario.
- **Metadata (Metadata API):** `title` (template `%s · proyecto-X`), `description`, `canonical`, `openGraph` (title/description/url/image/type), `twitter` (summary_large_image), `robots` (index,follow para el marketing). `metadataBase` configurado.
- **Datos estructurados (JSON-LD):** `Organization` + `WebSite` (con `potentialAction` SearchAction si aplica) en el layout; `SoftwareApplication` (la app, con `applicationCategory`, `operatingSystem`, rating placeholder) en la landing; `FAQPage` en support; `BreadcrumbList` donde aplique.
- **`sitemap.ts`** dinámico (todas las rutas públicas) y **`robots.ts`** (allow all + `sitemap` URL; la app de producto sería `noindex`, pero vive en otro dominio/host).
- **Canónicas** por página; **OG image** por defecto en `public/`.
- **Core Web Vitals:** `next/font` (fuente self-hosted/optimizada), `next/image` para imágenes, sin JS de cliente salvo lo imprescindible (componentes server por defecto).
- **Accesibilidad:** HTML semántico (`header/main/section/footer`, headings jerárquicos, `alt`), buen contraste.
- **18+ / cumplimiento:** mensajería de "18+" visible; enlaces a privacidad/términos; nada explícito en el marketing (apto para indexación y stores).

## 4. Contenido (shell — conciso, real pero no final)

- **Landing:** hero con propuesta de valor (app de citas/social geolocalizada LGBTQ+, 18+), grid de features (grid cercano, chat, álbumes privados, seguridad), sección "cómo funciona", callout de seguridad/privacidad, CTA a App Store / Google Play (enlaces placeholder), footer con legal/soporte/redes.
- **privacy / terms:** estructura con secciones reales (datos recogidos, derechos GDPR/CCPA, retención, contacto) — contenido placeholder serio, no "lorem".
- **support:** FAQ (cómo crear cuenta, seguridad, borrar cuenta, contacto) con JSON-LD FAQPage.
- **blog:** índice placeholder ("próximamente"), estructurado para SEO programático futuro (landings por ciudad, etc.).
