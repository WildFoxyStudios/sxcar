# F0.6 — Deploy & Infra (live provisioning) · Design

**Fecha:** 2026-06-26
**Fase:** F0.6 (fundación — despliegue/infra real)
**Depende de:** F0.1–F0.5 (todas integradas en `main`)

## 0. Objetivo
Levantar la infraestructura **real** y **coste-optimizada** para proyecto-X, con manejo seguro de secretos y todo como código (IaC) donde aplique. Arrancamos con **dominios temporales** y migramos a un dominio propio después.

Componentes:
- **Marketing (SEO):** `apps/marketing` → **Vercel** (proyecto nuevo `proyectox-marketing`), dominio `*.vercel.app` temporal.
- **Backend Rust API + Redis:** **AWS EC2** `t4g.small` (Graviton ARM) en **us-east-1**, Docker Compose (api + redis + reverse-proxy).
- **Postgres + PostGIS:** **Neon** (proyecto nuevo), connection string *pooled*.
- **Media:** **Cloudflare R2**, buckets nuevos `proyectox-media` / `proyectox-private` / `proyectox-verification`.

## 1. Decisiones del usuario (2026-06-26)
1. Crear recursos NUEVOS namespaced `proyectox-*` en las cuentas Cloudflare + Vercel ya conectadas (vía MCP). **No tocar** los 4 buckets ni los 30+ proyectos Vercel existentes.
2. AWS **us-east-1** + **t4g.small** (ARM, ~2 vCPU / 2 GB, ~$12/mes on-demand; candidato a Savings Plan/Reserved luego).
3. **Dominios temporales** primero (`.vercel.app` + DNS público de EC2). `site.url` placeholder se cambia en un commit al migrar.
4. Secretos: **yo creo los `.env`/`tfvars` (gitignored) con placeholders y el usuario rellena** Neon/AWS/R2. Nunca van a git ni al chat.

## 2. Acceso disponible vs requerido
- **Cloudflare:** MCP autenticado → puedo crear buckets R2 directamente. (Token S3 de R2 para el backend lo crea el usuario en el dashboard → va al `.env`.)
- **Vercel:** MCP autenticado (team `team_x1e9rnjCfTgJtV0f1G4sfzsq`) → puedo crear proyecto + deploy.
- **Neon:** sin MCP → el usuario crea el proyecto Neon y pega la connection string *pooled* en `infra/.env`.
- **AWS:** sin MCP → el usuario pone `AWS_ACCESS_KEY_ID`/`SECRET` (IAM con permisos EC2/VPC) en `infra/terraform/secrets.auto.tfvars` (gitignored), o ejecuta `terraform apply` él mismo.

## 3. Arquitectura de despliegue
```
                 Internet
                    │
        ┌───────────┴────────────┐
        │                        │
   Vercel CDN                AWS EC2 (us-east-1, t4g.small, ARM)
   proyectox-marketing       ┌──────────────────────────────┐
   (Next.js SSG, *.vercel.app)│ Docker Compose:              │
                              │  - caddy/nginx (reverse proxy)│
   Cloudflare R2              │  - api (Rust axum, :8080)     │
   proyectox-media/private/   │  - redis:7-alpine (:6379 int) │
   verification (S3 API)      └───────────────┬───────────────┘
        ▲                                     │ sqlx (TLS)
        │ presigned PUT/GET                   ▼
   mobile/web app  ───────────────────►  Neon (Postgres 16 + PostGIS, pooled)
```

### 3.1 Backend en EC2
- AMI: Ubuntu 24.04 LTS **arm64** (o Amazon Linux 2023 arm64). Docker + compose plugin.
- Imagen `api`: ya existe Dockerfile multi-stage (F0.1), digest-pinned, non-root. **Debe construirse para `linux/arm64`** (buildx) porque el VPS es Graviton.
- `redis:7-alpine` self-hosted (sin AOF persistente crítico; presencia/geo/rate-limit son efímeros). Solo en la red interna de compose, **nunca expuesto** a Internet.
- Reverse proxy: **Caddy**. Con dominio → auto-HTTPS (Let's Encrypt). **Sin dominio (temporal)** → Let's Encrypt NO emite para `*.compute.amazonaws.com` (sufijo público bloqueado) ⇒ la API sirve **HTTP** en `:80`/`:8080` sobre el DNS público de EC2 en staging. TLS llega con el dominio (subfase de migración). Documentado como limitación temporal de staging.
- Security Group: `22/tcp` solo desde la IP del usuario (`admin_cidr` en tfvars); `80/tcp` (y `443/tcp` para cuando haya dominio) público; **nada más**. Redis/Postgres NO expuestos.
- Secrets en el host: `/opt/proyectox/.env` (0600, root) con `DATABASE_URL` (Neon), `JWT_SECRET`, `REDIS_URL=redis://redis:6379`, claves R2. Inyectados a compose. Nunca en la AMI ni en git.

### 3.2 Neon (Postgres + PostGIS)
- Proyecto nuevo, región cercana a us-east-1 (`aws-us-east-1`/`aws-us-east-2`) para minimizar latencia con EC2.
- PostGIS: la migración `0001` hace `CREATE EXTENSION postgis` — Neon lo soporta. Verificar `vector`/`citext` si las usamos.
- Connection string **pooled** (PgBouncer, `-pooler` host) para la API; string **directo** para migraciones/DDL.
- Migraciones: `api migrate` (o `sqlx migrate run`) contra el string directo, desde local/CI con `DATABASE_URL` del `.env`. Free tier para arrancar; escalar a Launch/Scale luego.

### 3.3 Cloudflare R2
- 3 buckets nuevos: `proyectox-media` (renditions de perfil: blur+clear), `proyectox-private` (álbumes privados), `proyectox-verification` (selfies de verificación; retención corta, lifecycle).
- Acceso S3-compatible: el usuario crea un **R2 API Token** (scoped a esos buckets, Object Read & Write) → `R2_ACCOUNT_ID`/`R2_ACCESS_KEY_ID`/`R2_SECRET_ACCESS_KEY` al `.env`. El backend genera URLs presignadas; los clientes suben directo (coste de egress mínimo, sin pasar por EC2).
- CORS por bucket para subida directa desde mobile/web. (Exports GDPR → bucket `proyectox-exports` cuando construyamos esa feature.)

### 3.4 Vercel (marketing)
- Proyecto `proyectox-marketing`, root `apps/marketing`, framework Next.js. Build estático (ya verificado, 11 rutas ○).
- **Hardening pendiente de F0.5:** añadir CI/check de `npm run typecheck` (no existe job aún). En Vercel: el build no type-chequea (lleva `ignoreBuildErrors`), así que el gate de tipos lo da CI (GitHub Actions) o un `vercel.json`/`vercel.ts` que corra typecheck en build command. Decisión: `buildCommand = "npm run typecheck && npm run build"` para que Vercel falle si nuestros tipos rompen.
- Dominio `.vercel.app` temporal; `metadataBase`/`site.url` se actualizan al migrar.

## 4. Seguridad (pilar)
- 0 secretos en git: `.gitignore` cubre `**/.env`, `*.auto.tfvars`, `infra/terraform/*.tfstate*`, `.vercel/`. Plantillas `*.example` commiteadas.
- IAM mínimo (EC2/VPC), SG mínimo, Redis/PG no expuestos, SSH solo IP admin.
- Estado de Terraform: local + gitignored al arrancar (subfase: backend remoto en S3+DynamoDB lock para no perder estado / trabajo en equipo).
- R2 token scoped por bucket. JWT secret fuerte generado (no el de dev).
- App de producto (RN-web) seguirá `noindex` (no es este sitio).

## 5. Coste (pilar)
- EC2 t4g.small on-demand ~$12/mes (Graviton, barato); Redis local (sin coste gestionado); Neon free→pago por uso; R2 sin egress; Vercel Hobby/Pro. Optimización: una sola instancia para api+redis+proxy al inicio; separar cuando el tráfico lo exija.

## 6. Entregables / tareas (resumen; detalle en el plan)
1. `infra/` scaffolding + `.gitignore` + plantillas `.env.example` / `tfvars.example` + runbook.
2. **R2:** crear 3 buckets vía MCP (+ CORS) + documentar creación del token.
3. **Vercel:** crear proyecto `proyectox-marketing` + `vercel.json`/build command (typecheck+build) + deploy.
4. **Terraform AWS:** VPC mínima/SG + EC2 t4g.small + user-data (Docker, compose, pull imagen ARM, .env). `plan` revisable; `apply` cuando el usuario rellene creds.
5. **Neon:** runbook de creación + `.env` + correr migraciones + smoke (`/health` contra Neon).
6. **CI:** workflow marketing (typecheck/lint/test/build) + (scaffold) workflow build-and-deploy del backend ARM.
7. Smoke E2E: marketing live (SEO tags), API `/health` live (DB Neon ok), commit + actualizar ledger/memoria.

## 7. Riesgos
- TLS temporal sin dominio (API en HTTP en staging) → mitigado al migrar dominio (Caddy auto-HTTPS o Cloudflare delante).
- Build ARM: la imagen debe ser arm64 (buildx) o la instancia no la corre.
- Estado Terraform local → no perderlo (gitignored, backup); migrar a backend remoto pronto.
- Neon free tier: límites de cómputo/almacenamiento; escalar si hace falta.
- Secretos: dependen de que el usuario rellene `.env`; nada se aprovisiona en vivo en AWS/Neon hasta entonces.
